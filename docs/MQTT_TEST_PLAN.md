## Live-output validation procedure

The initial scale comparison is complete: the gateway percentage matched the
SANlight app after accounting for the app's whole-percent rounding. The procedure
below remains the regression method for additional profile points, lamp models and
firmware versions.

Use one lamp while its daily profile is stable:

1. Record `maxBrightness`, `liveBrightnessRaw`,
   `liveBrightnessPercentEstimate`, `lampClockSeconds` and `lampClock` after a
   manual read-only refresh.
2. Change MaxBrightness to a nearby safe value in `20..100` and wait for the
   verified write plus its live-status read.
3. Trigger another manual refresh and compare the raw live value before and
   after the change.
4. Restore the original MaxBrightness.
5. Where available, record smart-plug power at the same points as an independent
   relative comparison.

Expected working hypothesis:

```text
liveBrightnessRaw Ōēł daily-profile factor ├Ś MaxBrightness ├Ś 10
```

Do not treat the estimate as calibrated watts, photon flux or PPFD. Record the
raw value even when the percentage hypothesis is later revised. Avoid rapid
polling; each refresh sends Mesh queries and consumes sender Sequence Numbers.

## Current-output comparison ŌĆö 2026-07-17

A read-only refresh reported a raw vendor value of `334`, exposed as `33.4%` by
the gateway and native adapter. The SANlight app simultaneously displayed `34%`
for the same lamp. This confirms the `raw / 10` scale and shows that the app
rounds the display to a whole percentage. It does not establish a calibrated
relationship to electrical power, photon flux or PPFD.

## Clock-control validation for v0.3.0

Clock control remains manual and snapshot-based. There is no automatic synchronization,
drift marker, timezone policy or background clock polling.

For one lamp with its current clock recorded:

1. Trigger a read-only refresh and record `lampClockSeconds`, `lampClock` and
   `liveVerifiedAt`.
2. Trigger `refresh-gateway-info` and confirm that `localClockSeconds`,
   `localClock` and the gateway-info `timestamp` update without any Mesh command.
3. Apply a nearby arbitrary target time, wait for the verified write/readback result,
   and confirm that the observed lamp clock advances by only the expected execution time.
4. Trigger `sync-clock`, wait for verification, and compare the refreshed lamp snapshot
   with the gateway-local snapshot. Minute-level agreement is sufficient for this
   application; the gateway's internal write verification uses a few seconds of tolerance.
5. Restore the original or intended lamp time when the test used an artificial target.

For an all-lamps test, confirm that the result reports each node separately and that a
failed node does not hide successful nodes. Avoid repeating clock writes unnecessarily;
each write and readback consumes Mesh Sequence Numbers.

### Completed clock-control hardware run ŌĆö 2026-07-17/18

The v0.3.0 release candidate was exercised through the native adapter against
two real lamps. The run confirmed:

- canonical seconds/`HH:MM:SS` pairing in both directions;
- rejection and restoration for `24:00` and `86400`;
- verified single-lamp `sync-clock` and `set-clock`;
- verified all-lamp `sync-clock` and `set-clock` with per-node results;
- Mesh-free `refresh-gateway-info` without Sequence Number advancement;
- automatic reset of every one-shot ioBroker control to `false`;
- clean gateway and adapter logs for the clock commands;
- both lamp clocks restarting at `00:00:00` after a one-minute power cycle;
- verified all-lamp recovery synchronization after power returned; and
- fresh independent lamp reads within approximately one to two seconds of the
  host clock after recovery.

The final gateway wording commit passed 144 offline tests on the gateway Pi,
and `sanlight-gateway doctor` remained healthy. The all-lamp recovery command
took approximately 33 seconds, so clients must continue to treat `pending` as
an active command rather than a failure.

## Minimum regression sequence

Use the smallest test set justified by the changed files. Documentation-only
changes do not justify destructive or repeated lamp-state tests.

### 1. Repository and offline checks

```bash
git switch main
git pull --ff-only
git status --short
./scripts/run-tests.sh
bash scripts/install-gateway.sh --help
```

Expected: all tests pass, the worktree is clean and the public installer does
not expose `--reset-mesh-state`.

### 2. Reinstall/update the supported topology

```bash
sudo bash scripts/install-gateway.sh --reuse-existing
sudo sanlight-gateway doctor
sudo sanlight-gateway status
```

Expected: all three services are active and the doctor is healthy. No Mesh reset
or lamp write occurs.

### 3. Validate broker boundaries when installer code changes

Confirm:

- broker listens on the expected trusted-LAN interface/port;
- anonymous connection/publication is rejected;
- gateway credentials cannot publish commands outside their allowed direction;
- ioBroker credentials cannot write outside the exact command topic;
- credentials for one gateway ID cannot access another gateway root.

Do not print passwords in the test log.

### 4. Verify native adapter health

For the matching adapter instance, confirm:

```text
info.mqttConnected = true
info.gatewayOnline = true
info.protocolCompatible = true
info.connection = true
```

Check that retained lamp state is available below `lamps.<address>.state`.

### 5. Run a read-only refresh

Trigger `lamps.<address>.control.refresh` with `ack=false` and confirm:

```text
lamps.<address>.command.lastStatus = verified
```

This is the preferred end-to-end smoke test after documentation, packaging or
adapter-installation changes.

### 6. Run one reversible write only when write-path code changed

Record the verified current value, choose a nearby safe value in `20..100`, wait
for `verified`, independently check the lamp/app where practical, wait for the
write-rate guard, then restore the original value.

### 7. Re-run focused safety cases when affected code changes

Depending on the changed area, revalidate:

- duplicate command ID;
- live/offline retained command handling;
- expiry without Mesh traffic;
- same-node coalescing;
- persistent rate guard;
- blackout/restore;
- gateway and local broker restart recovery;
- full Raspberry Pi reboot;
- multi-instance isolation with a second physical gateway.

## Safety boundary

Do not repeat destructive Mesh resets, sequence recovery or blackout merely to
update documentation. Hardware claims require hardware evidence; unit tests
alone cannot prove RF, D-Bus timing, systemd ordering, broker behaviour or
physical lamp response.
